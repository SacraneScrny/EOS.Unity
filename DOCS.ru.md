# EOS — документация-рассказ. Часть 2: Unity-слой (EOS.Unity)

Это продолжение рассказа о фреймворке EOS. Первая часть — про движко-независимое
ядро — лежит в соседнем репозитории EOS, файл `DOCS.ru.md`. Без неё эту часть
читать можно, но тяжело: здесь постоянно упоминаются понятия из ядра (сущность,
компонент, система, инкарнация, командный буфер). Если какое-то слово непонятно —
в конце первой части есть словарик, в конце этой — свой, дополнительный.

Краткое напоминание сути: ядро EOS — чистый C# без единого упоминания Unity.
Оно умеет хранить сущности с компонентами, гонять системы и сохраняться, но само
по себе ничего не рисует и даже не знает, что такое «кадр» — кто-то снаружи должен
дёргать `Universe.Update(dt)`. Вот этот «кто-то снаружи» — и есть EOS.Unity.

Что делает этот слой, по пунктам:

1. **Крутит ядро из Unity** — встраивается в PlayerLoop и зовёт
   `Universe.Update/FixedUpdate/LateUpdate` каждый кадр.
2. **Даёт сущностям тела** — реализует «шов инкарнаций» ядра: по строковому
   идентификатору достаёт префаб и создаёт GameObject.
3. **Пулит вью** — переиспользует GameObject-ы вместо постоянных Instantiate/Destroy.
4. **Авторинг в редакторе** — ScriptableObject-пресеты: собрать сущность мышкой,
   а не кодом.
5. **Зеркалит иерархию** — родитель-ребёнок между сущностями отражается на их
   GameObject-ы, плюс компонент `EntityTransform` как «истина» о позиции.
6. **Сборки** — конструктор «корпус + модули в сокеты» (оружие на турели, колесо
   на машине).
7. **Кодогенерация запуска** — атрибуты `[EosBoot]`, `[EosWorldBootstrap]` и
   генерируемые файлы, чтобы старт игры не зависел от сцены.
8. **Редакторские окна** — инспектор живого мира, база инкарнаций и т.п.

Железное правило слоя: **ядро отсюда никогда не модифицируется**. Подключение
только через публичные швы: `Universe`, `IncarnationBridge`, `WorldBootstrap.Provider`,
`EosLog.OnLog`, `EosProfiler.Backend`, `WorldSerializer`/`WorldLoader`,
`world.ServiceRegistry`.

## 1. Как это лежит в проекте

EOS.Unity — это папка, которую кладут в `Assets/` Unity-проекта рядом с папкой
ядра. Раскладка по соглашению Unity:

```
EOS.Unity/
  Runtime/   → компилируется в Assembly-CSharp (основная сборка игры)
  Editor/    → компилируется в Assembly-CSharp-Editor (только редактор)
```

Здесь есть неочевидное решение, о которое спотыкаются: **asmdef-файлов нет
намеренно**. Asmdef (Assembly Definition) — это способ Unity выделить папку в
отдельную сборку. Ядро его не имеет, значит ядро попадает в общую сборку
`Assembly-CSharp` — и если бы у EOS.Unity был свой asmdef, он попросту не смог бы
ссылаться на типы из `Assembly-CSharp`. Поэтому оба слоя живут в одной сборке, и
у этого есть приятный побочный эффект: `internal`-члены ядра видны мосту (например,
пресеты добираются до записываемого мира по умолчанию, который наружу отдан только
для чтения). Если ядро когда-нибудь получит asmdef — этот пакет должен получить
свой тем же коммитом.

Прочие организационные мелочи:

- Внешняя зависимость одна: **MackySoft.SubclassSelector** — выпадающий список
  выбора типа компонента в инспекторе пресетов. Ставится через Package Manager
  (ссылка в `README.md`, раздел Requirements).
- `.gitignore` исключает `*.meta` — репозиторий вкладывается в Unity-проект,
  который генерирует свои меты.
- Тестов и команд сборки нет; ошибки компиляции всплывают в потребляющем проекте.
- Код в `Runtime/` не имеет права ссылаться на `UnityEditor`; всё редакторское —
  в `Editor/` или под `#if UNITY_EDITOR`.
- Все меню редактора — под `Sackrany ▸ EOS ▸ …`, меню создания ассетов — под
  `Sackrany/EOS/…`.

Быстрый старт (если хочется потрогать руками, прежде чем читать дальше) описан
в `README.md`, раздел Quick start: создать ресурсные папки → сгенерировать
бутстрап → положить префаб в `Resources/Incarnations/` → создать Entity Preset →
поставить в сцену Entity Preset Spawner → Play.

## 3. Запуск: от нажатия Play до первого кадра

Файлы: `Runtime/EosRuntimeBootstrap.cs`, `Runtime/EosLoop.cs`,
`Runtime/EosPlayerLoop.cs`, `Runtime/EosBootConfig.cs`,
`Editor/EosEditorTeardown.cs`, `Editor/BootstrapGeneratorWindow.cs`.

Запуск двухступенчатый: сначала **автоматическая** подготовка (происходит сама,
всегда), затем **явный** запуск (его вызываете вы или сгенерированный код).

### Ступень 1: EosRuntimeBootstrap — автоматика

`EosRuntimeBootstrap` помечен атрибутом Unity
`[RuntimeInitializeOnLoadMethod(SubsystemRegistration)]` — это значит «выполнить
очень рано, до загрузки первой сцены, при каждом входе в Play». Что он делает,
по порядку:

1. `EosDomainReset.Reset()` — сброс статики ядра (см. Часть 1, глава 20). Это
   критично для редактора с отключённой перезагрузкой домена (Enter Play Mode
   Options): без сброса статика прошлого запуска протекла бы в новый.
2. `UnityLogHandler.Install()` — логи ядра начинают попадать в консоль Unity.
3. `EosDomainResetRunner.Run()` — вызываются ваши методы с атрибутом
   `[EosDomainReset]` (свои статические поля тоже надо чистить — глава 4).
4. Регистрируются два встроенных байндера инкарнаций — `GameObjectBinder` и
   `EntityIncarnationBinder` (глава 5).
5. На `Application.quitting` подвешивается `EosLoop.Shutdown()` — корректный
   останов при выходе из игры.

Парный механизм в редакторе — `EosEditorTeardown` (`Editor/EosEditorTeardown.cs`):
останавливает EOS при выходе из Play Mode. Вместе эти двое дают «чисто вошли —
чисто вышли» в любом режиме редактора.

Заметьте: после ступени 1 EOS ещё **не запущен** — только подготовлен.

### Ступень 2: EosLoop.Boot — явный запуск

`EosLoop.Boot(config)` (`Runtime/EosLoop.cs:10`) — собственно включение. Метод
идемпотентен: повторный вызов при уже запущенном EOS — тихий выход. Порядок внутри:

```csharp
UnityLogHandler.MinLevel = config.MinLogLevel;  // фильтр логов
// профайлер: UnityProfilerBackend либо пустой бэкенд
IncarnationDatabase.Load();                     // индекс префабов (глава 5)
config.ApplyBinders();                          // ваши байндеры (каждый в try/catch)
Universe.Boot();                                // ядро! (+автозагрузка сейва)
EosPlayerLoop.Install();                        // встройка в кадр Unity
EosDebugDrawer.Ensure();                        // если config.DebugDraw
```

`Shutdown()` делает всё то же в обратном порядке плюс чистит пулы вью.

Настройки — `EosBootConfig` (`Runtime/EosBootConfig.cs`), сериализуемый класс,
который можно положить полем в MonoBehaviour и крутить в инспекторе:

```csharp
var config = new EosBootConfig
{
    EnableProfiler = false,   // замеры ядра в профайлер Unity
    DebugDraw = true,         // отладочная отрисовка гизмо
    MinLogLevel = LogLevel.Debug,
}
.AddBinder(new MyCustomBinder());  // свои байндеры — цепочкой

EosLoop.Boot(config);
```

Байндеры из конфига не регистрируются сразу — они запоминаются и применяются
внутри `Boot()`, после загрузки базы инкарнаций.

### EosPlayerLoop — как ядро получает кадры

PlayerLoop — это внутренний список этапов кадра Unity (Update, FixedUpdate,
рендер…), в который можно программно вставлять свои шаги. `EosPlayerLoop.Install()`
(`Runtime/EosPlayerLoop.cs`) вставляет три маркера — `EosFixedUpdate`, `EosUpdate`,
`EosLateUpdate` — в **начало** этапов `FixedUpdate`, `Update` и `PreLateUpdate`
соответственно. Каждый маркер просто зовёт соответствующий метод `Universe` с
дельтой времени Unity, под защитой `Universe.IsEnabled`.

«В начало этапа» означает: системы EOS отрабатывают **раньше** всех
MonoBehaviour-овских `Update()` в этом кадре. Установка идемпотентна — перед
вставкой удаляются свои же старые узлы, если остались.

### Кто зовёт Boot? GameBootstrap

Сам слой `EosLoop.Boot()` не вызывает — это ваша точка входа. Есть два пути:

- **Простой**: меню *Sackrany ▸ EOS ▸ Create Default Bootstrap*
  (`Editor/BootstrapGeneratorWindow.cs`) генерирует шаблон MonoBehaviour
  `GameBootstrap` с `[DefaultExecutionOrder(-10000)]` (выполняется раньше прочих
  скриптов) и сериализованным `EosBootConfig`. Кладёте его на объект в первой
  сцене — готово.
- **Продвинутый, без сцены**: атрибутный запуск, следующая глава.

## 4. Атрибуты запуска и кодогенерация

Файлы: `Runtime/Boot/*.cs`, `Editor/Boot/EosBootCodegen.cs`,
`Editor/Boot/EosWorldBootstrapCodegen.cs`, `Editor/Boot/EosSystemRegistryCodegen.cs`.

Проблема, которую решает эта глава: «как сделать, чтобы мой код инициализации
выполнялся при старте игры, не завися от того, какая сцена загружена первой и
что в ней лежит». Решение EOS — атрибуты на статических методах + генерируемые
файлы.

### Общий принцип: «включается существованием»

Все три генерируемых файла живут в `Assets/EOS.Generated/` и подчиняются одному
правилу:

- **Создание — только вручную**, через пункт меню. Код их никогда не создаёт сам.
- **Пока файл существует**, он автоматически перегенерируется при каждой
  перекомпиляции скриптов (через `[DidReloadScripts]`; запись на диск — только
  если содержимое реально изменилось, чтобы не дёргать Unity лишний раз).
- **Удалили файл — выключили механизм.**
- Методы неправильной формы пропускаются с предупреждением в лог — сгенерированный
  файл всегда компилируется.

### EosBootstrap.gen.cs — шаги старта игры

Меню: *Sackrany ▸ EOS ▸ Create Auto Bootstrap*. Генератор —
`Editor/Boot/EosBootCodegen.cs`. Собирает по проекту два вида методов:

```csharp
public static class GameInit
{
    [EosBootConfigProvider]                       // настроить конфиг
    public static EosBootConfig Configure(EosBootConfig c)
    {
        c.EnableProfiler = true;
        return c.AddBinder(new MyBinder());
    }

    [EosBoot]                                     // шаг после Boot
    public static void RegisterSaveHooks() { ... }

    [EosBoot(order: 10)]                          // позже нулевых
    public static void WarmCaches() { ... }
}
```

Требования к форме (см. XML-доки в `Runtime/Boot/EosBootAttribute.cs`):
`[EosBoot]` — `public static void` без параметров; `[EosBootConfigProvider]` —
`public static EosBootConfig(EosBootConfig)`; тип — публичный недженериковый.

Сгенерированный `Run()` выполняется на `RuntimeInitializeOnLoadMethod(BeforeSceneLoad)`
(чуть позже SubsystemRegistration, всё ещё до первой сцены) и имеет две ветки:

- **Холодный путь** (EOS ещё не запущен): провайдеры по очереди пропускают через
  себя один конфиг → `EosLoop.Boot(config)` → все `[EosBoot]`-шаги по порядку.
  С этим файлом `GameBootstrap` в сцене вообще не нужен.
- **Тёплый путь** (кто-то уже запустил EOS раньше): выполняются только шаги с
  `isFallback: true` — на случай, когда часть инициализации нужна в обоих
  сценариях.

Порядок шагов: сначала явные рёбра `[EosBootBefore(typeof(X))]` /
`[EosBootAfter(typeof(X), method)]` (топологическая сортировка), при равенстве —
`Order`, дальше имя типа/метода. Цикл зависимостей не валит генерацию — пишется
в лог, и рёбра цикла понижаются до сортировки по `Order`. Каждый шаг в рантайме
обёрнут в собственный `try/catch` — один упавший шаг не срывает остальные.

### WorldBootstrap.gen.cs — засев каждого мира

Меню: *Sackrany ▸ EOS ▸ Create World Bootstrap*. Генератор —
`Editor/Boot/EosWorldBootstrapCodegen.cs`. Собирает методы вида:

```csharp
[EosWorldBootstrap]
public static void SetupWorld(World world)
{
    world.ServiceRegistry.Register(new PathfindingService());
    world.Context.Set(new GameSettings { Difficulty = 1f });
}
```

Сгенерированный `Install()` срабатывает на `SubsystemRegistration` и записывает
сборный метод в шов ядра `WorldBootstrap.Provider` (Часть 1, глава 16). Дальше
ядро само: **каждый** `World.Init()` и **каждый** `World.Reset()` прогоняет мир
через эти методы. Сортировка — по `Order`, потом по имени (графа Before/After
здесь нет).

Разница с `[EosBoot]`, которую важно прочувствовать: `[EosBoot]` — «один раз за
сессию игры», `[EosWorldBootstrap]` — «для каждого мира, включая пересоздание и
сброс». Сервисы и дефолты контекста — сюда; подписки на глобальные хуки — туда.

### EosGeneratedSystems.gen.cs — быстрые системы

Меню: *Sackrany ▸ EOS ▸ Create System Registry*. Генератор —
`Editor/Boot/EosSystemRegistryCodegen.cs` — это просто редакторская обвязка над
генератором из ядра (`SystemRegistryGenerator`, Часть 1, глава 10): получает
текст и записывает с той же логикой «существует — держим свежим». Пока файл на
месте, реестр пересобирается при каждой перекомпиляции и никогда не отстаёт от
ваших систем; удалили — ядро вернулось на рефлексию. Это чисто оптимизация:
поведение идентично, меняется только скорость и количество мусора в кадре.

### [EosDomainReset] — третий атрибут, без кодогенерации

```csharp
static int _killCount;   // статическое поле — переживает Play Mode без домен-релоада!

[EosDomainReset]
static void ResetStatics() => _killCount = 0;
```

Методы с этим атрибутом находит **рефлексией** `EosDomainResetRunner`
(`Runtime/Boot/EosDomainResetRunner.cs`) и вызывает в начале каждой игровой
сессии (шаг 3 из главы 3). Форма — `static void()` любой видимости, порядок не
гарантируется. Зачем это нужно: при отключённой перезагрузке домена ваши
статические поля сохраняют значения между нажатиями Play — этот атрибут даёт
место, где их чистить.

Ловушка IL2CPP: раз вызовы идут через рефлексию, строгая очистка кода может
вырезать такие методы из сборки. Помечайте их `[UnityEngine.Scripting.Preserve]`.
(Сгенерированные пути `[EosBoot]`/`[EosWorldBootstrap]` зовутся обычным кодом —
им это не грозит.)

## 5. Инкарнации: как сущность получает GameObject

Файлы: `Runtime/Incarnation/IncarnationIndex.cs`, `IncarnationDatabase.cs`,
`GameObjectBinder.cs`, `EntityIncarnationBinder.cs`, `EntityIncarnation.cs`;
со стороны редактора — `Editor/IncarnationIndexBuilder.cs`,
`Editor/IncarnationDatabaseWindow.cs`.

Вспомним шов из ядра (Часть 1, глава 17): компонент `Incarnation<TView>` хранит
строковый идентификатор и просит зарегистрированный байндер «создай мне вью».
Здесь этот шов наконец обретает плоть.

### Откуда берутся префабы: индекс инкарнаций

Идентификатор инкарнации — это **путь к префабу** под `Resources/Incarnations/`
без расширения. Префаб `Assets/Resources/Incarnations/Units/Knight.prefab` имеет
идентификатор `"Units/Knight"`.

Соответствие «идентификатор → путь» лежит в файле `Resources/incarnations.json`
(класс `IncarnationIndex`: списки `Entries { Id, Path }` и
`Redirects { OldId, NewId }`). Этот файл **не пишется руками** — его пересобирает
редактор: `IncarnationIndexBuilder` (AssetPostprocessor) реагирует на любые
изменения префабов в этой папке, плюс есть меню *Sackrany ▸ EOS ▸ Rebuild
Incarnation Index*.

Зачем нужны `Redirects`: идентификаторы попадают в сейвы. Переименовали или
перенесли префаб — старые сохранения держат старый идентификатор. Запись-редирект
`{ OldId: "Knight", NewId: "Units/Knight" }` чинит это: при разрешении
идентификатора редиректы прослеживаются по цепочке (с защитой от зацикливания —
максимум 64 перехода). Пересборщик индекса заметит переименование, напишет в лог
готовый к вставке сниппет редиректа и — важно — **не затирает** ваши ручные
редиректы при пересборках.

В рантайме всем этим заведует `IncarnationDatabase`: `Load()` (лениво) читает
JSON, `Resolve(id)` следует редиректам и кэширует загруженные префабы,
`Unload()` чистится при останове. Посмотреть содержимое индекса глазами — окно
*Sackrany ▸ EOS ▸ Incarnation Database*.

### Байндеры: два встроенных

Оба регистрируются автоматически на ступени 1 запуска (глава 3).

**`GameObjectBinder`** (`IIncarnationBinder<GameObject>`) — простейший:
заспавнить префаб и забыть. Никакой связи с сущностью, пустые синхронизации.
Подходит для чистой декорации:

```csharp
entity.Add<Incarnation<GameObject>>().Setup("Props/Barrel");
```

**`EntityIncarnationBinder`** (`IIncarnationBinder<EntityIncarnation>`) — рабочая
лошадка. Ожидает на корне префаба компонент-наследник `EntityIncarnation`:

```csharp
public class KnightView : EntityIncarnation
{
    protected override void OnBind()   { /* связали с сущностью; Entity доступна */ }
    protected override void OnUnbind() { /* сейчас отвяжут */ }
    protected override void OnSync()      { /* каждый Update */ }
    protected override void OnSyncFixed() { /* каждый FixedUpdate */ }
    protected override void OnSyncLate()  { /* каждый LateUpdate */ }
}
```

`EntityIncarnation` (`Runtime/Incarnation/EntityIncarnation.cs`) — абстрактный
MonoBehaviour со свойством `Entity` — мостик «из мира GameObject-ов обратно к
данным». Внутри `OnSync` обычно читают компоненты сущности и обновляют визуал:

```csharp
protected override void OnSync()
{
    if (Entity.TryGet<Health>(out var hp))
        _healthBar.fillAmount = hp.Value / 100f;
}
```

Жизненный цикл со стороны байндера: создать экземпляр → присвоить `view.Entity` →
`OnBind`; при уничтожении — `OnUnbind` → очистить `Entity` → деспавн. Все
пользовательские хуки обёрнуты в `try/catch` — кривой `OnSync` не уронит общий
проход синхронизации.

Кто зовёт синхронизации: системы ядра `IncarnationSync*System` (Часть 1, глава 17)
каждый кадр дёргают `Sync/SyncFixed/SyncLate` всех инкарнаций, а байндер
переадресует их в `OnSync/OnSyncFixed/OnSyncLate` вашей вьюхи.

И ключевая деталь напоследок: **оба байндера спавнят и деспавнят не напрямую, а
через `ViewPoolRegistry`** — поэтому пулинг (следующая глава) включается на уровне
префаба и совершенно прозрачен для байндеров и для вашего кода.

## 6. Пул вью

Файлы: `Runtime/Pooling/IncarnationPooling.cs`, `ViewPoolRegistry.cs`,
`GameObjectPool.cs`, `PooledView.cs`, `IPoolableView.cs`.

`Instantiate` и `Destroy` — дорогие операции. Если префаб создаётся часто (пули,
эффекты, всплывающие цифры), его экземпляры выгодно переиспользовать: «умерший»
объект не уничтожать, а выключить и спрятать, при следующем спавне — достать и
включить. Это и есть пул.

**Включение — на префабе**: добавьте на корень компонент `IncarnationPooling`
с двумя настройками: `Preload` (сколько экземпляров создать заранее, ≥ 0)
и `MaxSize` (потолок хранимых в пуле, по умолчанию 32; лишние при возврате
уничтожаются по-настоящему). Нет компонента — нет пула, обычный Instantiate/Destroy.

Как это работает внутри. `ViewPoolRegistry.Spawn(prefab)` смотрит на префаб:
помечен — арендует из `GameObjectPool` этого префаба, нет — инстанцирует.
`Despawn(instance)` пытается вернуть экземпляр его пулу (каждый пуловый экземпляр
несёт автоматически добавленный маркер `PooledView` со ссылкой на свой пул);
не вышло (не пуловый, пул переполнен) — уничтожает. Отдыхающие экземпляры лежат
выключенными под скрытым корневым объектом `"EOS View Pool"`. Реестр пулов
чистится при `EosLoop.Shutdown()` и через `[EosDomainReset]`.

Главное, что нужно знать пользователю пула: **пул не сбрасывает состояние
экземпляра**. Unity не вызывает заново `Awake`/`Start` у реактивированного
объекта — погасший трейл, запущенная корутина, изменённый цвет останутся от
прошлой жизни. Для сброса есть интерфейс:

```csharp
public class TrailResetter : MonoBehaviour, IPoolableView
{
    public void OnRent()   { _trail.Clear(); }  // достали из пула
    public void OnReturn() { _audio.Stop(); }   // возвращают в пул
}
```

`OnRent`/`OnReturn` вызываются на **всех** компонентах экземпляра, реализующих
интерфейс, включая детей (рекурсивно, каждый в `try/catch`). Порядок аккуратный:
при аренде объект сначала отцепляется от пула и активируется, потом приходит
`OnRent`; при возврате сначала `OnReturn`, потом деактивация и парковка.

Контраст с пулом *компонентов* из ядра (Часть 1, глава 8) — намеренный: там
жизненный цикл (`Awake`/`Start`) честно проигрывается заново, поэтому хуков нет;
здесь Unity жизненный цикл не повторяет, поэтому хуки есть.

## 7. Пресеты: сборка сущностей в редакторе

Файлы: `Runtime/Presets/EntityPreset.cs`, `EosCloneUtility.cs`,
`EntityComponentSet.cs`, `ComponentSetBuilder.cs`, `ComponentSetModule.cs`,
`EntityPresetSpawner.cs`; редактор — `Editor/Presets/*.cs`.

До сих пор сущности собирались кодом: `new EosEntity(...)`, `Add<...>()`,
`Setup(...)`. Пресеты позволяют описать всё то же самое ассетом в редакторе.

### EntityPreset

Создание: *Assets → Create → Sackrany → EOS → Entity Preset*. Внутри ассета:

- имя сущности, флаги «активна при спавне» и «сериализуема»;
- вид инкарнации (`None` / `EntityIncarnation` / `GameObject`) и идентификатор —
  в инспекторе это выпадающий список из индекса инкарнаций, не строка вслепую;
- теги;
- **шаблоны компонентов** — список настроенных экземпляров `EosObject` прямо
  в ассете (поле с `[SerializeReference]`; выбор типа — через выпадающее дерево
  по пространствам имён, спасибо SubclassSelector);
- ссылки на **наборы компонентов** (ниже) и переопределения к ним;
- **модули по умолчанию** — для сборок (глава 9).

Использование:

```csharp
var entity = preset.Instantiate();          // в мир по умолчанию
var entity2 = preset.Instantiate(myWorld);  // в конкретный мир
```

Порядок работы `Instantiate` (см. `Runtime/Presets/EntityPreset.cs:52`) стоит знать,
потому что он отвечает на вопрос «кто кого переопределит»:

1. Сущность создаётся **неактивной** — вся сборка происходит «при выключенном
   питании», и компоненты получат `Awake` уже на полностью собранной сущности
   (см. Часть 1, глава 5 — зачем сущности рождаются выключенными).
2. Применяются наборы компонентов, по порядку; переопределения пресета побеждают
   по совпадению типа.
3. Теги.
4. Инкарнация: `entity.Add<Incarnation<TView>>().Setup(id)`.
5. Собственные компоненты пресета.
6. Если в пресете стоит галка Active — `entity.On()`.
7. Модули по умолчанию (спавн и присоединение дочерних пресетов в сокеты).

Если на любом шаге вылетело исключение — ошибка в лог, недособранная сущность
уничтожается, возвращается `EosEntity.Null`: половинчатых сущностей в мире не
бывает.

### EosCloneUtility — почему «шаблон», а не «образец напрямую»

Компонент в ассете — это один экземпляр. Сущностей по пресету — много. Значит,
при каждом спавне шаблон нужно **скопировать** в свежий компонент. Этим занимается
`EosCloneUtility.CopyDeclaredFields` (`Runtime/Presets/EosCloneUtility.cs`), и
у него есть принцип, который объясняет его странности: **копируется ровно то,
что сериализовала бы Unity**. Публичные поля и поля с `[SerializeField]` /
`[SerializeReference]` — да; статические, readonly, `[NonSerialized]` — нет.
Массивы, `List<>`, `[Serializable]`-классы и структуры копируются вглубь;
примитивы, строки и ссылки на Unity-объекты — как есть; делегаты и коллекции из
`System.Collections` — отказ. Глубина ограничена 32 уровнями (дальше —
предупреждение).

Логика принципа: если поле не пережило бы сохранение ассета в Unity, оно не должно
«как бы работать» при спавне — иначе вы получите поведение, которое сломается при
первом перезапуске редактора. Честное «нет» сразу лучше загадочного «иногда».

### EntityComponentSet — переиспользуемые куски

*Assets → Create → Sackrany → EOS → Component Set* — это «полуфабрикат»: набор
тегов и шаблонов компонентов, который подключают несколько пресетов сразу
(условный набор «Юнит»: Health + Movement + тег Unit — один на рыцаря, лучника
и голема). Помимо списка, у набора может быть **кодовый модуль** — наследник
`ComponentSetModule` с методом `Build(ComponentSetBuilder)`: способ добавить
компоненты кодом там, где списка недостаточно.

Правило одного компонента: на сущности один компонент каждого типа. Наборы
применяются по порядку, добавления пресета — после них; если нужно поменять
*значения* в компоненте из набора — в инспекторе пресета жмётся Override на
блоке набора, и появляется локальная переопределяющая копия (Revert возвращает).

### EntityPresetSpawner — мостик в сцену

MonoBehaviour (`Runtime/Presets/EntityPresetSpawner.cs`): ссылка на пресет,
галка `_spawnOnStart` (по умолчанию включена — спавнит в `Start`), галка
`_destroyAfterSpawn` (уничтожить себя после спавна — «одноразовая точка спавна»),
свойство `LastSpawned`. Можно звать `Spawn()` вручную.

Важно: спавнер **требует уже запущенного EOS** (`EosLoop.IsBooted`) и сам ничего
не запускает — иначе ошибка в лог. Порядок обеспечивается тем, что бутстрап
(глава 3) выполняется либо до сцены вовсе, либо с `[DefaultExecutionOrder(-10000)]`.

### Инспекторы

`Editor/Presets/EntityPresetEditor.cs` и `EntityComponentSetEditor.cs` — это та
самая «магия» в инспекторе: сворачиваемые блоки Info/Data с запоминанием состояния
на ассет, кнопки Override/Revert/Delete на блоках, выпадающий список
идентификаторов инкарнаций из индекса, выпадающий список сокетов для модулей по
умолчанию (читается прямо из `SocketSet` префаба) и кнопка Spawn в Play Mode для
быстрой проверки. `ComponentPickerDropdown` — дерево выбора типа компонента,
`PresetEditorUtility` — общая рисовалка этих блоков.

## 8. Иерархия вью и EntityTransform

Файлы: `Runtime/Hierarchy/HierarchyViewSystem.cs`, `EntityTransform.cs`,
`EntityTransformSyncSystem.cs`.

В ядре есть иерархия сущностей (Часть 1, глава 6). У сущностей есть вью. Вопрос:
если сущность-меч прицеплена к сущности-рыцарю, должен ли GameObject меча ездить
за GameObject-ом рыцаря? Эта глава отвечает «да» — двумя системами, работающими
каждый кадр.

### HierarchyViewSystem — зеркало родительских связей

Для каждой сущности с вью, у которой есть родитель в иерархии сущностей, система
перевешивает её GameObject под GameObject **ближайшего предка, у которого есть
вью** (`SetParent(..., worldPositionStays: true)` — мировая поза сохраняется).
Безвьюшные промежуточные сущности прозрачны: если меч → рука → рыцарь, а у «руки»
вью нет, GameObject меча повиснет прямо под рыцарем. Вью корневых сущностей система
не трогает. На событие `ParentChanged` (отцепление) вью возвращается в корень сцены —
но только если она всё ещё висит под вью старого родителя (вдруг вы уже перевесили
её сами).

Система «самозалечивающаяся»: она не разово реагирует, а каждый кадр приводит
вью-иерархию к соответствию с сущностной. Работает в `IncarnationGroup` до
синхронизации инкарнаций.

### EntityTransform — позиция как данные ECS

Одного зеркала связей мало, если поза должна быть частью **данных** (например,
сохраняться в сейв). Для этого компонент:

```csharp
public sealed class EntityTransform : EosObject, IObjectSerializable, IPoolableObject
{
    public Vector3 LocalPosition;     // локальная поза —
    public Quaternion LocalRotation;  // относительно родителя-сущности
    public Vector3 LocalScale;
    public Vector3 WorldPosition { get; set; }   // мировые — пересчёт по иерархии
    public Quaternion WorldRotation { get; set; }
    public Vector3 LossyScale { get; }
}
```

Это **единственное авторитетное место** локальной позы сущности. Мировые свойства
складывают позу по цепочке предков (предки без компонента считаются «тождеством» —
как будто стоят в нуле без поворота; масштаб перемножается покомпонентно — с теми
же огрублениями, что lossyScale в Unity). Статические `GetWorldTrs` /
`GetTrsRelativeTo` — общая математика этой композиции, ей же пользуются сборки.

### EntityTransformSyncSystem — данные → вью

Каждый Update применяет компонент к GameObject-у: если вью висит под вью предка —
ставится локальная поза (просуммированная через безвьюшных промежуточных), если
вью в корне сцены — мировая. Перед записью значения сравниваются со старыми, чтобы
не помечать неизменённые трансформы грязными.

Принципиально: **данные текут в одну сторону, ECS → вью**. Хотите подвинуть
объект — меняйте `EntityTransform`, не `transform` GameObject-а: ручную правку
трансформа система перезапишет на следующем кадре.

Слои подключаются по отдельности: только иерархия (без `EntityTransform`) — вью
ездят за родителями с сохранением мировой позы; добавили `EntityTransform` — поза
стала авторитетной со стороны ECS и попала в сохранения.

## 9. Сборки: сокеты и модули

Файлы: `Runtime/Assembly/*.cs`, `Editor/Assembly/ModuleKindFieldDrawer.cs`.
Историческая спецификация — `EOS.UnityAssembly.md` (учтите: локальные смещения
с тех пор переехали в `EntityTransform`).

Сборка — это конструктор: у сущности-«корпуса» есть именованные **сокеты**
(гнёзда), в которые вставляются сущности-**модули**. Танк с гнёздами «башня» и
«двигатель», робот с гнёздами на руки. От голой иерархии (глава 8) это отличается
**типизацией**: сокет принимает только модуль подходящего **вида**, гнездо может
быть занято только одним модулем, и вся конструкция сериализуема.

При этом сборка — **надстройка над нативной иерархией ядра**: присоединение модуля
заодно делает его обычным ребёнком сущности-родителя. Поэтому каскадное
уничтожение, наследование активности и `GetParent` работают для модулей сами собой.

### Словарь сборки

- **`ModuleKind`** (`Runtime/Assembly/ModuleKind.cs`) — «вид модуля»: лёгкая
  структура (интернированный номер + имя), создаётся `ModuleKind.Of("Weapon")`
  или `ModuleKind.Of<MyEnum>(...)`. Сериализуется **по имени** — сейвы переживают
  перестановку значений. `ModuleKindCatalog` (ScriptableObject) + атрибут
  `[ModuleKindField]` на строковом поле дают в инспекторе выпадающий список видов
  по всем каталогам проекта (с запасным свободным вводом).
- **`SocketSet`** (`Runtime/Assembly/SocketSet.cs`) — MonoBehaviour на корне
  **префаба инкарнации** родителя. Список сокетов: `Id` (имя гнезда), `Kind`
  (какой вид модулей принимает), `Anchor` (Transform — точка крепления вью).
  С гизмо в редакторе. То есть сокеты — свойство визуального тела, и находятся
  они в рантайме через вью сущности.
- Компоненты на сущностях (все сериализуемы и пулятся, с чисткой полей в
  `OnDispose`): **`EntityAssembly`** на родителе (карта «сокет → ребёнок»;
  в `OnDispose` каскадно уничтожает свои модули), **`Module`** на ребёнке (его
  вид), **`AttachedTo`** на ребёнке (кто родитель, в каком сокете; плюс
  рантайм-флаги).

### AssemblyService — операции

Один сервис на мир; регистрируется через `[EosWorldBootstrap]`
(`Runtime/Assembly/AssemblyServiceBootstrap.cs`), а `world.Assemblies()` создаст
его лениво, даже если кодогенерация world bootstrap не включена. Удобнее всего —
методы-расширения (`Runtime/Assembly/AssemblyExtensions.cs`):

```csharp
weapon.AttachTo(tank, "TurretSocket");                    // присоединить
weapon.AttachTo(tank, "TurretSocket", localPos, localRot); // с локальным смещением
weapon.DetachFromSocket();                                 // отсоединить
tank.TryGetModule("TurretSocket", out var module);
world.Assemblies().IsSocketFree(tank, "TurretSocket");
world.Assemblies().GetModules(tank, listToFill);
```

`Attach` проверяет: сокет свободен, модуль ни к кому не присоединён, цикл не
образуется — и тогда записывает связь с обеих сторон и ставит нативного родителя.
Вариант со смещением пишет его в `EntityTransform` модуля; вариант без — не трогает
существующий. `Detach` снимает `AttachedTo` и отвязывает нативного родителя
(только если тот всё ещё указывает на сборочного родителя). Обе операции публикуют
EOS-события `ModuleAttached` / `ModuleDetached` (`Runtime/Assembly/AssemblyEvents.cs`) —
на них можно реагировать `EventExecute`-методом.

Тонкость с серединой кадра: присоединение/отсоединение — структурные изменения
(добавляются/удаляются компоненты), которые посреди итерации запрещены ядром.
Расширения решают это сами: вне итерации — выполняют немедленно, изнутри систем —
откладывают на `world.AfterCurrentPhase` с повторной проверкой условий при
выполнении. Вам думать не надо, просто знайте, что из системы эффект наступит
в конце фазы.

### Вью модуля: AssemblyViewBindSystem

Связали сущности — а GameObject-ы? `AssemblyViewBindSystem`
(`Runtime/Assembly/AssemblyViewBindSystem.cs`) каждый кадр находит присоединённые,
но ещё не «привязанные по вью» модули и ждёт, пока существуют **обе** вью (вью
создаются асинхронно по меркам кадра). Когда обе на месте: проверяется совпадение
вида модуля с видом сокета (несовпадение — отсоединение с ошибкой), вью модуля
перевешивается под `Anchor` сокета, локальная поза берётся из `EntityTransform`
(нет компонента — тождество), флаг `ViewBound` ставится.

Та же система слушает `ParentChanged` и **сверяет две правды**: если нативный
родитель модуля больше не совпадает с `AttachedTo.Parent` (кто-то сделал модулю
`entity.Detach()` или `SetParent` мимо сборки), сокетная связь снимается — сборка
самозалечивается за один тик. Немедленный путь — честный `DetachFromSocket()`.

### Модули по умолчанию

В пресете (глава 7) можно прописать список `DefaultModule { SocketId, EntityPreset }`:
при спавне пресета в каждый указанный сокет автоматически заспавнится и
присоединится дочерний пресет (рекурсивно, с пределом глубины 32; не присоединилось —
сирота уничтожается). Семантика «только для новых»: при восстановлении из сейва
модули по умолчанию **не** спавнятся — сейв сам помнит, что и где было присоединено
(а чего уже не было — оторванную руку робота сейв не отрастит обратно).

### Именование, чтобы не путаться

`entity.Detach()` — из ядра, отвязка от нативного родителя.
`module.DetachFromSocket()` — отсюда, отсоединение из сборки (которое заодно
корректно снимет нативную связь). Расширения с одинаковым именем `Detach` в двух
пространствах имён давали неоднозначный вызов — поэтому имя сборочного нарочно
другое.

## 10. Сохранения в Unity

Своего формата сохранений слой не навязывает — он лишь пользуется механикой ядра
(Часть 1, глава 18) и гарантирует, что всё его хозяйство в неё вписано:

- `Incarnation<TView>` сериализует идентификатор → после загрузки вью пересоздаются
  из префабов (через те же байндеры и пулы).
- Редиректы индекса (глава 5) чинят идентификаторы из старых сейвов после
  переименований префабов.
- Иерархия едет в `EntityRecord.ParentLocalId`, поза — в данных `EntityTransform`,
  сборочные связи — в данных `EntityAssembly` / `AttachedTo` (обе стороны).
- Подключение хранилища — ваши пара строк в `[EosBoot]`-шаге:

```csharp
[EosBoot]
public static void SetupSaves()
{
    WorldLoader.OnSave = snapshot => File.WriteAllText(path, MySerializer.ToJson(snapshot));
    WorldLoader.OnLoad = () => File.Exists(path) ? MySerializer.FromJson(File.ReadAllText(path)) : null;
}
```

После этого `WorldSerializer.Save()` пишет сейв, а `Universe.Boot()` сам подхватит
его при следующем запуске. Снимок — простые объекты; формат (JSON, бинарный,
что угодно) — на ваш вкус.

Что **не** сохраняется: позы вью без `EntityTransform` (вью встанет в позу префаба,
мировая поза сохранится при перецеплении), содержимое пулов, всё с выключенными
флагами сериализуемости.

## 11. Логи, профилировщик, отладочная отрисовка

Файлы: `Runtime/UnityLogHandler.cs`, `Runtime/UnityProfilerBackend.cs`,
`Runtime/EosDebugDrawer.cs`.

Три маленьких моста к трём швам ядра:

- **`UnityLogHandler`** подписывается на `EosLog.OnLog` и пишет в консоль Unity
  в виде `[EOS:Источник] сообщение`; уровень important — `LogError`/`LogWarning`/
  `Log` по уровню записи. `MinLevel` отфильтровывает лишнее ещё до форматирования.
- **`UnityProfilerBackend`** превращает `Begin(label)`/`End()` ядра в
  `ProfilerMarker`-ы Unity (метки видны в окне Profiler). Маркеры кэшируются по
  метке; стек следит за парностью, лишний `End()` безопасен. Включается флагом
  `EnableProfiler` конфига.
- **`EosDebugDrawer`** — скрытый служебный GameObject, чей `OnDrawGizmos` зовёт
  `Universe.DebugDraw()` — так ваши переопределения `OnDebugDraw` в компонентах
  и системах получают законный контекст для `Gizmos.Draw*`. Только в Play Mode;
  создаётся при `DebugDraw = true`.

## 12. Редакторские инструменты

Файлы: `Editor/EosFolderBootstrap.cs`, `Editor/Visualization/*.cs` и упомянутые ранее.

- **`EosFolderBootstrap`** — при загрузке редактора (и по меню *Create EOS
  Resources*) создаёт недостающие папки `Assets/Resources`,
  `Resources/Incarnations`, `Resources/EntityPresets` и пустой
  `ModuleKindCatalog.asset`, если в проекте нет ни одного. Трогает диск только
  когда чего-то нет.
- **World Inspector** (*Sackrany ▸ EOS ▸ World Inspector*,
  `Editor/Visualization/EosInspectorWindow.cs`) — главный инструмент «понять, что
  происходит в мире». Три вкладки: **Live** — живые сущности с компонентами;
  **Systems** — системы, их группы, порядок и фазы; **Groups & Archetypes** —
  обзорный граф. Холст (`EosGraphView`) умеет панораму/зум/перетаскивание. Данные
  обновляются ~10 раз в секунду опросом мира только на чтение (`EosIntrospection`);
  закрытое окно не стоит ничего — никаких статических хуков в рантайме. Кнопка
  Copy dump использует `WorldDebug` из ядра. Учтите: сопоставление
  «архетип ↔ система» приближённое — интерфейсные параметры и теговые фильтры
  не учитываются, это намеренное упрощение визуализации.
- **Incarnation Database** — глава 5; **генераторы** — глава 4;
  **инспекторы пресетов** — глава 7.

## 13. Грабли: собрание известных ловушек

Сводный список вещей, которые «работают не так, как ожидаешь» — каждая упоминалась
по ходу рассказа, здесь — чтобы было где искать, когда что-то загадочно сломалось.

1. **Генерируемые файлы не создаются сами.** Только меню. Но раз созданные —
   обновляются при каждой перекомпиляции; чтобы выключить механизм, файл удаляют.
2. **Сборочные операции из систем — только отложенно.** Расширения сами откладывают,
   но прямые вызовы `AssemblyService` посреди итерации словят исключение от
   `StructuralChangePolicy` ядра.
3. **Симметрия Spawn/Despawn.** Всё, что заспавнено байндерами, идёт через
   `ViewPoolRegistry`. Меняя код байндеров, сохраняйте парность — иначе пуловые
   префабы либо утекут, либо словят двойной Destroy.
4. **Пул не чистит состояние вью.** Сброс — ваша работа в
   `IPoolableView.OnRent/OnReturn` (глава 6).
5. **Клонирование пресетов = сериализация Unity.** Поле, которое Unity не
   сохранила бы в ассет, не скопируется и при спавне (глава 7).
6. **`[EosDomainReset]` + IL2CPP** — методы зовутся рефлексией и могут быть
   вырезаны строгой очисткой кода; помечайте `[Preserve]`. `[EosBoot]` и
   `[EosWorldBootstrap]` в безопасности — их зовёт сгенерированный код.
7. **Спавнер не запускает EOS.** `EntityPresetSpawner` требует готового
   `EosLoop.IsBooted`; бутстрап должен отработать раньше.
8. **Идентификатор инкарнации — путь.** Переименовали префаб — добавьте редирект
   в `incarnations.json`, иначе старые сейвы не найдут вью (глава 5).
9. **Гонка «отцепил и убил в одном кадре».** Уничтожение каскадно идёт
   «дети-сначала», и это позволяет детским вью вернуться в пулы до смерти
   родительского GameObject-а. Но если в одном кадре отцепить ребёнка и уничтожить
   его бывшего родителя, вью-системы не успеют перевесить вью ребёнка — она умрёт
   вместе с родительской. Отцепляйте кадром раньше либо перевешивайте вью вручную.
10. **Поза в сейве — только с `EntityTransform`.** Без него вью после загрузки
    встаёт в позу префаба (глава 8 и 10).
11. **`EntityTransform` авторитетен.** Ручная правка локального трансформа
    прицепленной вью перезатрётся на следующем Update-синке. Двигайте компонент,
    а не GameObject.

## 14. Словарик второй части

(Базовые термины — в словарике Части 1.)

- **PlayerLoop** — внутренний список этапов кадра Unity, в который можно
  программно вставлять свои шаги.
- **`RuntimeInitializeOnLoadMethod`** — атрибут Unity «выполнить метод при старте
  игры» (этап указывается: `SubsystemRegistration` — самый ранний, до сцены;
  `BeforeSceneLoad` — перед первой сценой).
- **Перезагрузка домена (domain reload)** — пересоздание всего .NET-окружения
  при входе в Play Mode; сбрасывает статические поля. В Unity её можно отключить
  ради скорости — тогда статику надо чистить руками (`[EosDomainReset]`).
- **asmdef (Assembly Definition)** — способ Unity выделить папку кода в отдельную
  сборку. Здесь не используется намеренно (глава 1).
- **ScriptableObject** — ассет с данными в Unity-проекте; на нём построены
  пресеты, наборы компонентов и каталоги видов модулей.
- **`[SerializeReference]`** — атрибут Unity, позволяющий сериализовать в ассете
  поле полиморфно (хранить наследника по ссылке) — так шаблоны компонентов лежат
  в пресете.
- **Префаб** — заготовка GameObject-а в ассетах, из которой создаются экземпляры.
- **Инкарнация** (напоминание) — визуальное воплощение сущности; идентификатор =
  путь к префабу под `Resources/Incarnations/`.
- **Редирект** — запись «старый идентификатор → новый» в индексе инкарнаций,
  чинит сейвы после переименований.
- **Пул** — хранилище выключенных экземпляров для переиспользования вместо
  создания/уничтожения.
- **Сокет** — именованное типизированное гнездо на префабе вью, куда крепится
  модуль; **модуль** — сущность с видом, вставляемая в сокет; **вид модуля
  (ModuleKind)** — типизирующая метка, по которой сокет принимает или отвергает
  модуль.
- **Якорь (Anchor)** — Transform на префабе, к которому физически крепится вью
  модуля.
- **TRS** — позиция, поворот, масштаб (Translation, Rotation, Scale) как единое
  «положение объекта».
- **Кодовый модуль набора (`ComponentSetModule`)** — класс, добавляющий компоненты
  в набор программно, в дополнение к списку в ассете.
- **IL2CPP** — компилятор Unity «C# → C++» для релизных сборок; его очистка кода
  (stripping) вырезает «неиспользуемые» методы — опасно для всего, что зовётся
  рефлексией.
- **Гизмо (gizmos)** — отладочная графика поверх сцены в редакторе
  (`OnDrawGizmos`).
